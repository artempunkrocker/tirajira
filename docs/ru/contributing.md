# Участие в разработке

Благодарим вас за интерес к развитию TiraJira! Мы приветствуем вклад сообщества в виде сообщений об ошибках, предложений по улучшению и пулл-реквестов.

## Как внести свой вклад

### Сообщения об ошибках

Если вы нашли ошибку в работе TiraJira:

1. Проверьте, не сообщалось ли уже об этой ошибке в [issues](https://github.com/artempunkrocker/tirajira/issues)
2. Если нет, создайте новое issue с подробным описанием:
   - Версия TiraJira
   - Версия Python
   - Операционная система
   - Шаги для воспроизведения ошибки
   - Ожидаемое и фактическое поведение
   - Любая дополнительная информация, которая может помочь

### Предложения по улучшению

Если у вас есть идея для нового функционала или улучшения:

1. Проверьте [issues](https://github.com/artempunkrocker/tirajira/issues) на наличие похожих предложений
2. Если нет, создайте новое issue с описанием:
   - Что вы хотите добавить или улучшить
   - Почему это полезно
   - Как это может быть реализовано (если есть идеи)

### Пулл-реквесты

Перед началом работы над пулл-реквестом:

1. Создайте issue с описанием планируемых изменений
2. Получите одобрение от мейнтейнеров
3. Форкните репозиторий
4. Создайте ветку для ваших изменений
5. Внесите изменения
6. Напишите тесты (если применимо)
7. Обновите документацию (если применимо)
8. Отправьте пулл-реквест

## Настройка среды разработки

1. Форкните и клонируйте репозиторий:
   ```bash
   git clone https://github.com/ваш-логин/tirajira.git
   cd tirajira
   ```

2. Установите Poetry (если еще не установлен):
   ```bash
   pip3 install poetry
   ```

3. Установите зависимости:
   ```bash
   poetry install
   ```

4. Активируйте виртуальное окружение:
   ```bash
   poetry shell
   ```

## Структура кода

- `src/tirajira/` - Основной исходный код
  - `auth/` - Аутентификация в Jira
  - `commands/` - Команды CLI
  - `file_loaders/` - Загрузчики файлов различных форматов
  - `report_writers/` - Генераторы отчетов
  - `utils/` - Вспомогательные функции
- `tests/` - Тесты
- `docs/` - Документация
- `examples/` - Примеры использования

## Стиль кода

Мы следуем стандартам кодирования Python:

- Форматирование кода с помощью Ruff
- Документирование функций и классов в стиле Google Python Style Guide
- Использование type hints
- Английский язык для имен переменных и комментариев
- Русский язык для docstrings (документация для пользователей)

### Проверка кода

Перед отправкой пулл-реквеста убедитесь, что:

1. Код проходит форматирование:
   ```bash
   ruff check .
   ```

2. Все тесты проходят:
   ```bash
   pytest
   ```

3. Покрытие кода не ниже 90%:
   ```bash
   pytest --cov=.
   ```

## Написание тестов

- Все новые функции должны сопровождаться тестами
- Используйте pytest для написания тестов
- Используйте моки для внешних зависимостей (Jira API, файловая система)
- Тесты должны проверять как нормальные случаи, так и ошибочные ситуации
- Следуйте структуре директорий в `tests/`, соответствующей `src/tirajira/`

Пример теста:
```python
def test_create_task_success(mocker):
    """Тест успешного создания задачи"""
    # Подготовка мока Jira клиента
    mock_jira = mocker.Mock()
    mock_jira.create_issue.return_value = mocker.Mock(key="PROJ-123")
    
    # Вызов тестируемой функции
    result = create_task(mock_jira, {"project": {"key": "PROJ"}, "summary": "Test"})
    
    # Проверка результата
    assert result.success is True
    assert result.issue_key == "PROJ-123"
```

## Документация

- Все публичные функции, классы и методы должны иметь docstrings на русском языке
- Обновляйте документацию при изменении функциональности
- Добавляйте примеры использования в `examples/` при добавлении новых возможностей

## Коммиты

Мы используем conventional commits для автоматической генерации changelog:

- `feat:` - Новый функционал
- `fix:` - Исправление ошибок
- `docs:` - Изменения в документации
- `style:` - Изменения, не влияющие на смысл кода (форматирование, пробелы и т.д.)
- `refactor:` - Рефакторинг кода
- `test:` - Добавление или изменение тестов
- `chore:` - Обслуживание (обновление зависимостей и т.д.)

Примеры:
```
feat: добавлена поддержка XML формата файлов
fix: исправлена ошибка при создании подзадач
docs: обновлена документация по аутентификации
```

## Процесс релизов

Релизы создаются автоматически при мерже в основную ветку:

1. Изменения мержатся в `main`
2. Semantic Release анализирует коммиты
3. При наличии подходящих изменений создается новый релиз
4. Пакет публикуется в PyPI
